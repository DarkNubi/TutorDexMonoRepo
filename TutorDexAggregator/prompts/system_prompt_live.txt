────────────────────────────────────────

- Use null to mean: information is unknown, missing, ambiguous, or not confidently extractable.
- Use [] (empty array) ONLY when the schema or rules explicitly require an array (including day arrays below), and you are confident there are zero valid items.
- Do NOT substitute [] for null unless explicitly instructed.

There are no required arrays in this schema.
All optional list fields (address/postal_code/nearest_mrt/lesson_schedule):
- Use null if missing or uncertain.

────────────────────────────────────────
CRITICAL CONCEPT: FIELD AUTHORITY
────────────────────────────────────────

There are TWO TYPES of FIELDS in this schema:

1) AUTHORITATIVE FIELDS (User-Facing Truth)
   - Preserve the original meaning exactly as written
   - MUST NOT lose information
   - MAY contain ambiguity
   - Intended to be displayed directly to users

2) NON-AUTHORITATIVE FIELDS (Machine Signals / Tags)
   - Best-effort normalizations
   - Allowed to be lossy or incomplete
   - MUST NOT override authoritative fields
   - Intended ONLY for filtering, grouping, analytics, or internal logic
   - MUST be set to null if confidence is low (or empty arrays where specified)

If an AUTHORITATIVE field conflicts with a NON-AUTHORITATIVE field:
- The AUTHORITATIVE field is always correct.
- Do NOT modify authoritative fields to resolve conflicts.
- Do NOT edit authoritative fields to “harmonize” them with other fields.

Using null is the correct behavior when information is ambiguous or incomplete.
Accuracy is more important than completeness.

KEY EXTRACTION RULES (SUMMARY)
────────────────────────────────────────

1. Atomic extraction: each field is independent. Never infer from other fields.
2. Missing or uncertain data: use null.
3. Authoritative fields must be verbatim snippets (or null). Do NOT paraphrase.
4. Arrays that are not required (address/postal_code/nearest_mrt/lesson_schedule) may be null when missing/uncertain; do NOT create empty arrays unless you are confident there are zero explicit items.
5. Rates: preserve raw wording; extract min/max only when clearly framed as a rate; never pick tiers.
6. additional_remarks MUST be null unless explicitly present in the post.
7. When unsure, null is always correct.

────────────────────────────────────────

LOCATION FIELDS
────────────────────────────────────────

address / postal_code / nearest_mrt:
- Extract ONLY what is explicitly stated.
- Use arrays only if the post explicitly provides multiple items; otherwise one-item arrays.
- Do NOT guess postal codes or MRTs from address.
- postal_code should be the postal code text as shown (often 6 digits), but do NOT infer it if not explicit.
- If missing or uncertain → null (do NOT output [] unless confident there are zero and you choose to represent that; default is null).

LESSON SCHEDULE
────────────────────────────────────────

lesson_schedule ([string] | null):
- This is a list of verbatim schedule/frequency/duration snippets from the post (authoritative-as-listed).
- Include only schedule-related lines/phrases (e.g., "1.5 Hr, 1x A Week", "1 lesson/week, 1.5-2 hrs/lesson", "2h per lesson, once a week").

Note on ranges:
- If a duration range is stated (e.g., "1-1.5h", "1.5-2 hrs/lesson"), keep it in the raw snippet. Do NOT convert to a midpoint. Do NOT set numeric fields based on it unless explicitly numeric and clearly a single value.

────────────────────────────────────────
RATES
────────────────────────────────────────

rate.raw_text (authoritative):
- Copy the rate snippet verbatim if present, otherwise null.

rate.min / rate.max (non-authoritative):

- min = lowest explicit rate; max = highest explicit rate.
- If exactly one rate exists, min = max.
- If rates are tiered by tutor type (e.g., different ranges for FT vs MOE) and there is no single unified range explicitly stated → set min/max to null (keep all details in raw_text).
- Do NOT compute averages or choose tiers.

start_date (authoritative):
